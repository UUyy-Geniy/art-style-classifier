# **Техническое задание и база знаний: Art Style Classifier**

## **1\. Обзор проекта**

**Art Style Classifier** — это высоконагруженная ML-система, предназначенная для автоматической классификации художественных стилей изображений. Система построена по асинхронной архитектуре с разделением контуров выполнения (Data Plane) и управления (Control Plane).

## **2\. Требования к системе (MoSCoW)**

| Приоритет | Функциональность |
| :---- | :---- |
| **Must Have** | Загрузка фото, распознавание широкого спектра стилей, просмотр топ-предсказаний с вероятностями, дообучение модели на новых данных, горячая замена моделей. |
| **Should Have** | Авторизация/регистрация пользователей, просмотр истории личных запросов, пакетная загрузка нескольких фото, просмотр справочной информации по стилям. |
| **Could Have** | Глобальная статистика популярных стилей, сбор фидбэка пользователей для дообучения (Data Flywheel). |

## **3\. Архитектура контейнеров (C4 Level 2\)**

Система состоит из следующих независимых компонентов:

* **Web Application (React/SPA):** Интерфейс пользователя для взаимодействия с системой.  
* **Core API (FastAPI):** Центральный узел бизнес-логики, валидации и координации задач.  
* **Message Broker (Redis):** Шина обмена сообщениями (очередь задач и Pub/Sub уведомления).  
* **Inference Worker (PyTorch):** Вычислительный узел для прогона нейросетей на GPU.  
* **PostgreSQL:** Реляционная база для хранения профилей, истории и справочников.  
* **Object Storage (S3):** Хранилище оригиналов изображений.  
* **Model Registry (MLflow):** Система версионирования весов и артефактов моделей.

## **4\. Жизненный цикл данных (Claim Check Pattern)**

Для обеспечения масштабируемости и стабильности используется паттерн передачи "квитанции" вместо передачи файлов через брокер:

1. **Загрузка:** Web App \-\> Core API (POST запрос с файлом).  
2. **Сохранение:** Core API \-\> S3 (оригинал сохраняется, API получает путь).  
3. **Постановка задачи:** Core API \-\> Redis (Task Queue). Передается JSON: {"task\_id": "uuid", "s3\_path": "path/to/img"}.  
4. **Обработка:** Inference Worker вычитывает задачу из Redis, скачивает файл из S3, выполняет инференс.  
5. **Запись результата:** Inference Worker \-\> PostgreSQL (сохранение класса и уверенности).  
6. **Уведомление:** Inference Worker \-\> Redis (Event Channel). Публикация сообщения о готовности.  
7. **Выдача:** Core API получает сигнал и уведомляет Web App (через WebSockets или Polling).

## **5\. Спецификации компонентов для реализации**

### **5.1 Core API (FastAPI)**

* **Эндпоинты:**  
  * POST /v1/upload — прием файла, асинхронный ответ с task\_id.  
  * GET /v1/status/{task\_id} — проверка готовности.  
  * GET /v1/history — личная история (требует JWT).  
  * POST /v1/feedback — прием оценки пользователя.

### **5.2 Inference Worker (ML Engine)**

* **Стек:** Python, PyTorch, Docker (с поддержкой nvidia-container-runtime).  
* **Логика:** Singleton-загрузка модели из Model Registry при старте. Бесконечный цикл прослушивания Redis (BRPOP).

### **5.3 База данных (PostgreSQL)**

Основные таблицы: Users, Styles (справочник), Predictions (id, user\_id, style\_id, s3\_path, confidence, user\_feedback, created\_at).

## **6\. MLOps и администрирование**

* **Реестр моделей:** Используется MLflow. Веса хранятся в S3, метаданные — в PostgreSQL.  
* **Обновление моделей:** Реализовано через тегирование версий в MLflow (Staging/Production).  
* **Контур дообучения:** Внешние скрипты собирают пары "Image \+ Feedback" из БД и S3, обучают новую версию и пушат в MLflow.

## **7\. Рекомендации по развертыванию (DevOps)**

* **Изоляция GPU:** Inference Worker должен запускаться в изолированном контейнере с пробросом GPU.  
* **Масштабирование:** Web API масштабируется по CPU/RAM, Inference Worker — горизонтально по количеству доступных GPU-карт.  
* **Безопасность:** S3 и PostgreSQL должны быть закрыты во внутреннем контуре, доступ к ним имеет только API и Worker.
